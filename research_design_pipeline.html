01 The three concerns · addressed

Three things
you said.
Three things this fixes.

Every concern raised in the design workshop maps to a specific automation in this pipeline. The intent isn't to add research — it's to take the operational weight off the three of you.

Research will slow shipping.
What changes

Async tests, 24-hour turnaround

Maze runs unmoderated tests on every prototype with 20–40 testers. No scheduling, no designer time, results back inside the same sprint. Faster than waiting for the next moderated session.

It eats designer bandwidth.
What changes

Designers judge results, not run ops

Zapier handles the test launch. Notion stores the task scripts. Claude synthesizes the output. The only human steps that remain are the ones that actually need a designer's judgment.

My research never reaches the roadmap.
What changes

Auto-write to Vistaly + Jira

Every test result becomes a Vistaly card and a Jira ticket with source citations. Posts to #research. Same repository as PM research — your work is finally cross-referenced with theirs.

03 Phase 01 · in detail

Sprint discovery.
What the team already knows.

When a designer creates a Jira ticket for a new feature area, the system automatically pulls relevant past research and surfaces unanswered questions. You open Figma already loaded with context.

i.
Phase 01 · Trigger: Jira ticket created

Sprint kickoff with context

Jira ticket → Vistaly lookup + Question bank scan → Slack #design-sprint
  • Vistaly lookup queries past research tagged to the relevant feature area. If the team already learned something about this surface, you see it before designing.
  • Question bank (Notion) returns the 3–5 unanswered questions that match the ticket's tags. These become the brief for any test you run this sprint.
  • Slack digest posts both into #design-sprint the moment the ticket is created — the designer opens Figma already knowing what's known and what's not.
For the designer

~5 minutes of context instead of an afternoon of digging through old Figma files and Slack threads. You start designing with research in the room.

04 Phase 02 · in detail

Prototype testing.
One field. Three sources.

The core of the pipeline. The designer pastes one URL into one Jira field — regardless of whether the prototype is built in Figma, deployed via Cursor + Claude to Vercel, or exported as a Claude artifact to Netlify. The automation handles the rest.

One trigger. One synthesis engine. Three outputs that feed the roadmap.
a.
Phase 02 · Sub-system

Three prototype sources, one Jira field

The designer doesn't have to think about plumbing. Whatever you built, paste the URL into the Prototype URL field on the Jira ticket. Zapier reads the URL pattern and routes to the correct Maze template.

  • Figma prototype (figma.com/proto/…) — share the full prototype URL, not a single frame. Covers multi-frame flows. Routes to Maze's click-through template.
  • Vercel preview (*.vercel.app) — for prototypes built with Cursor + Claude. Real working code, real interactions, real error states. Routes to the web prototype template.
  • Netlify deploy for Claude artifacts. Artifacts are session-private — one netlify drop command turns them into a public URL. Same web prototype template as Vercel.
For the designer

You build however you build. Paste, walk away, results in 24 hours.

b.
Phase 02 · Sub-system

The task script — the step that's usually missing

Without a task script, a Maze test is a prototype demo, not a usability test. Two paths cover every case:

  • Pre-built scripts for the 5 known critical flows: onboarding, first diagram, AI chat, export, upgrade. Stored in Notion. Zapier fetches the matching one based on the ticket's flow tag.
  • Claude-drafted scripts for new flows. Claude reads the Jira ticket context and drafts 3 task statements. Posted to Slack for a 30-second human review before Maze fires.
  • Tasks are injected into Maze alongside the prototype URL in a single API call. No copy-paste, no tab switching.
For the designer

You don't write task scripts unless the flow is genuinely new. And even then, the 30-second review is the only thing required.

05 Phase 03 · in detail

Post-ship
validation.
Catch regressions early.

Once a feature ships, the same Maze template runs again — this time on the live product. Compares to the prototype baseline. Catches the things that change between prototype and production.

iii.
Phase 03 · Trigger: Jira → Done

The same test, on the live thing

Jira: Done → Maze re-run on live URL + Typeform triggers activate → Claude · Vistaly · Slack
  • Same Maze template runs on the live feature URL. Comparing against the prototype baseline highlights anything that drifted between design and engineering.
  • Typeform behavioral surveys activate on the new feature: 2 questions on first use, 1 question on a return visit, 1 on abandonment. Continuous signal between moderated sessions.
  • Same synthesis through Claude → Vistaly → Slack. The post-ship card sits next to the pre-ship card in Vistaly so the comparison is visible without effort.
For the designer

Predefined prompts, coldstart templates, AI chat — features with zero adoption — stop being mysteries. You see the post-ship reality within 24 hours of shipping, not in the next quarterly review.

06 Deep dive · Cursor + Claude prototypes

When the prototype
can also be the
handoff.

The Vercel-preview path is worth its own section because it changes how prototypes work in this team. A coded prototype isn't just a fancier Figma — it's a different kind of test, and a different relationship with engineering.

Why coded prototypes

Real inputs.
Real states.
Real handoff.

Build the prototype in Cursor with Claude as a pair. Push to a Vercel preview URL. Maze tests it like any other web prototype — except now the tester is interacting with actual code, not a click-through illusion. They can type into real fields, hit real error states, see real loading patterns.

Richer test signal

  • Testers encounter actual interactions, not preset clicks
  • Edge cases surface that Figma click-throughs hide
  • Loading, error, and empty states get tested honestly
  • Form validation gets real-typed input

Becomes the dev handoff

  • Engineering gets working code, not a Figma file to rebuild
  • Design tokens, spacing, animations all encoded
  • The prototype repository is the spec
  • Cuts the design → engineering rebuild step entirely
07 Before & after · for designers

What changes
for the three of you.

Today

Gut-led, isolated, lost.

  • Design starts with a gut call — no research surfaced, no question bank
  • Validation happens after the design is conceived, not before
  • Setup overhead for any test eats designer hours
  • Findings live in Figma, sometimes Notion — almost never reach Vistaly
  • Research never crosses paths with PM or Growth findings
  • Zero-adoption features only become visible months after ship
After

Context-loaded, async, in the roadmap.

  • Sprint kickoff comes with prior research and open questions in Slack
  • Validation runs in parallel with design, not after
  • Setup is a single Jira field — no manual scheduling, no template writing
  • Every test result auto-creates a Vistaly card and a Jira ticket
  • Same repository as PM & Growth — findings are cross-referenced
  • Post-ship Maze catches regressions within 24 hours
