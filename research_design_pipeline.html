Every concern raised in the design workshop still maps to a specific automation in this pipeline. The Great Question swap reshapes the middle of Phase 02 but the destination is the same — less operational weight on the three of you.

Async tests, 24-hour turnaround

03 Phase 01 · in detail

Sprint discovery.
What the team already knows.

When a designer creates a Jira ticket for a new feature area, the system automatically pulls relevant past research and surfaces unanswered questions. You open Figma already loaded with context.

i.
Phase 01 · Trigger: Jira ticket created

Sprint kickoff with context

Jira ticket → Vistaly + GQ repo lookup + Question bank scan → Slack #design-sprint
  • Vistaly + GQ repo lookup queries past research tagged to the relevant feature area, across both the Vistaly card layer and Great Question's native insights and highlights. If the team already learned something about this surface, you see it before designing.
  • Question bank (Notion) returns the 3–5 unanswered questions that match the ticket's tags. These become the brief for any test you run this sprint.
  • Slack digest posts both into #design-sprint the moment the ticket is created — the designer opens Figma already knowing what's known and what's not.
For the designer

~5 minutes of context instead of an afternoon of digging through old Figma files and Slack threads. You start designing with research in the room.

04 Phase 02 · in detail

Prototype testing.
One template. One URL. Publish.

The core of the pipeline. The designer takes whatever prototype they built — Figma, Vercel preview, Netlify drop, GitHub Pages — opens the matching Great Question template, pastes the URL into the website-test block, hits publish. The rest runs on its own.

Five templates. One synthesis engine. Three outputs that feed the roadmap.
a.
Phase 02 · Sub-system

Three prototype sources, five GQ templates

The designer doesn't have to think about plumbing. Whatever you built, the workflow is the same: open the matching Great Question template, paste the URL into the website-test block, hit publish. GQ accepts any hosted URL — it doesn't care what generated it.

  • Figma prototype (figma.com/proto/…) — share the full prototype URL, not a single frame. Covers multi-frame flows. Drops straight into the GQ website-test block.
  • Vercel preview (*.vercel.app) — for prototypes built with Cursor + Claude. Real working code, real interactions, real error states.
  • Netlify drop / GitHub Pages for Claude artifacts. Artifacts are session-private — one netlify drop or one push to gh-pages turns them into a public URL. GQ accepts both.
For the designer

You build however you build. Open template, paste URL, publish — results in 24 hours.

b.
Phase 02 · Sub-system

The task script — already written, inside the template

Without a task script, an unmoderated test is a prototype demo, not a usability test. The five pre-baked GQ templates already have their scripts written for the common flows:

  • Five pre-baked templates covering onboarding, first diagram, AI chat, export and share, plan upgrade. Each lives in GQ as a draft study with research goal, screener, incentive, and task block all set. The only thing missing is the URL. The five drafts already exist in the workspace.
  • Claude-drafted scripts for new flows. When the flow is genuinely new, Claude reads the Jira ticket context and drafts 3 task statements. A fresh GQ unmoderated study takes ~10 minutes to assemble from those — including the website-test block.
  • One field changes per test. Once the template is open, only the prototype URL changes. Same recordings, same transcripts, same Claude synthesis on the way out.
For the designer

You don't write task scripts for the five common flows ever. New flows take ~10 minutes, once.

c.
Phase 02 · Honest gap

What's not automated yet — and why

v1 of this pipeline promised "paste URL into Jira, walk away." That assumed Maze's API would accept URL + task script in one shot. GQ's public surfaces today (MCP and Zapier) don't expose study-block editing — the only mutating actions available are study shell, screener, and candidate management. So URL patching is manual until either GQ ships the endpoint or we add a thin URL redirector layer.

  • What's still automatic: recruitment, screening, scheduling, recording, transcript capture, Claude synthesis, Vistaly card creation, Jira ticket creation, Slack posting.
  • What's manual: opening the GQ template and pasting the prototype URL into the website-test block. ~2 minutes per test.
  • Path forward (optional): a thin URL redirector (Cloudflare Worker or Netlify redirect) lets each template point at a stable URL we control — restores the v1 "paste in Jira, walk away" promise. Half a day of engineering, deferred until we feel the manual cost.
05 Phase 03 · in detail

Post-ship
validation.
Catch regressions early.

Once a feature ships, the same Great Question template runs again — this time pointed at the live product URL. Compares to the prototype baseline. Catches the things that change between prototype and production.

iii.
Phase 03 · Trigger: Jira → Done

The same test, on the live thing

Jira: Done → GQ template re-run on live URL + Typeform triggers activate → Claude · Vistaly · Slack
  • Same GQ template runs on the live feature URL. Comparing against the prototype baseline highlights anything that drifted between design and engineering.
  • Typeform behavioral surveys activate on the new feature: 2 questions on first use, 1 question on a return visit, 1 on abandonment. Continuous signal between moderated sessions.
  • Same synthesis through Claude → Vistaly → Slack. The post-ship card sits next to the pre-ship card in Vistaly so the comparison is visible without effort. GQ's own repo holds the transcripts and highlights side-by-side too.
For the designer

Predefined prompts, coldstart templates, AI chat — features with zero adoption — stop being mysteries. You see the post-ship reality within 24 hours of shipping, not in the next quarterly review.

06 Deep dive · Cursor + Claude prototypes

When the prototype
can also be the
handoff.

The Vercel-preview path is worth its own section because it changes how prototypes work in this team. A coded prototype isn't just a fancier Figma — it's a different kind of test, and a different relationship with engineering.

Why coded prototypes

Real inputs.
Real states.
Real handoff.

Build the prototype in Cursor with Claude as a pair. Push to a Vercel preview URL. Great Question tests it like any other web prototype — except now the tester is interacting with actual code, not a click-through illusion. They can type into real fields, hit real error states, see real loading patterns.

Richer test signal

  • Testers encounter actual interactions, not preset clicks
  • Edge cases surface that Figma click-throughs hide
  • Loading, error, and empty states get tested honestly
  • Form validation gets real-typed input

Becomes the dev handoff

  • Engineering gets working code, not a Figma file to rebuild
  • Design tokens, spacing, animations all encoded
  • The prototype repository is the spec
  • Cuts the design → engineering rebuild step entirely
07 Before & after · for designers

What changes
for the three of you.

Today

Gut-led, isolated, lost.

  • Design starts with a gut call — no research surfaced, no question bank
  • Validation happens after the design is conceived, not before
  • Setup overhead for any test eats designer hours
  • Findings live in Figma, sometimes Notion — almost never reach Vistaly
  • Research never crosses paths with PM or Growth findings
  • Zero-adoption features only become visible months after ship
After

Context-loaded, async, in the roadmap.

  • Sprint kickoff comes with prior research and open questions in Slack
  • Validation runs in parallel with design, not after
  • Setup is opening one GQ template, pasting one URL — no scheduling, no script writing
  • Every test result auto-creates a Vistaly card, a GQ insight, and a Jira ticket
  • Same repository as PM & Growth — findings are cross-referenced
  • Post-ship GQ test catches regressions within 24 hours
